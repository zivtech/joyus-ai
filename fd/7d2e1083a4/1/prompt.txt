- **File:** `/Users/AlexUA/claude/joyus-ai/joyus-ai-state/src/enforcement/audit/index.ts:196-207`

### [HIGH] Unbounded HTTP body read enables memory exhaustion in IPC server

- **File:** `/Users/AlexUA/claude/joyus-ai/joyus-ai-state/src/service/ipc.ts:52-54`
- **Category:** bug / security
- **Description:** The `/capture` endpoint accumulates all incoming chunks into a string with no size cap.       res.writeHead(413);
      res.end('Payload Too Large');

    }
  }

### 1. [CRITICAL] Weak Encryption: CryptoJS AES with Passphrase Instead of Key
- **File:** `/Users/AlexUA/claude/joyus-ai/joyus-ai-mcp-server/src/db/encryption.ts:21`
- **Category:** A02 — Cryptographic Failures
- **Exploitability:** Local with database access; remote if DB is compromised
- **Blast Radius:** All stored OAuth tokens (Google, Jira, Slack, GitHub) for every user are exposed. An attacker with database read access obtains full API access to every connected service for every user.
- **Description:** `CryptoJS.AES.encrypt(token, ENCRYPTION_KEY)` uses a **string passphrase**, not a raw key. CryptoJS interprets a string second argument as a passphrase and derives a key via a single-iteration `EvpKDF` (MD5-based, no salt randomness), which is trivially brutable. Additionally, the IV is derived deterministically from the passphrase, meaning identical plaintexts produce predictable ciphertext patterns. This is not a production-grade encryption scheme.
- **Remediation:**
```typescript
// BAD — CryptoJS passphrase mode (single-iteration MD5 KDF)
import CryptoJS from 'crypto-js';
return CryptoJS.AES.encrypt(token, ENCRYPTION_KEY).toString();

// GOOD — Node.js native crypto with AES-256-GCM, random IV, proper key
import crypto from 'node:crypto';

const ALGORITHM = 'aes-256-gcm';
const IV_LENGTH = 16;
const TAG_LENGTH = 16;

export function encryptToken(token: string): string {
  const key = Buffer.from(process.env.TOKEN_ENCRYPTION_KEY!, 'hex'); // 32 bytes
  const iv = crypto.randomBytes(IV_LENGTH);
  const cipher = crypto.createCipheriv(ALGORITHM, key, iv);
  const encrypted = Buffer.concat([cipher.update(token, 'utf8'), cipher.final()]);
  const tag = cipher.getAuthTag();
  // Store as: iv:tag:ciphertext (all hex)
  return `${iv.toString('hex')}:${tag.toString('hex')}:${encrypted.toString('hex')}`;
}

export function decryptToken(stored: string): string {
  const key = Buffer.from(process.env.TOKEN_ENCRYPTION_KEY!, 'hex');
  const [ivHex, tagHex, encHex] = stored.split(':');
  const decipher = crypto.createDecipheriv(ALGORITHM, key, Buffer.from(ivHex, 'hex'));
  decipher.setAuthTag(Buffer.from(tagHex, 'hex'));
  return decipher.update(Buffer.from(encHex, 'hex')) + decipher.final('utf8');
}
```

## Architecture Review: joyus-ai

## Summary

The joyus-ai repository contains two well-separated TypeScript packages and a Python library with no cross-package import dependencies. The architecture demonstrates strong domain separation through PostgreSQL schema namespacing (`public`, `content`, `pipelines`) and clean interface-based abstractions for pluggable components (connectors, search providers, entitlement resolvers, voice analyzers). However, there are several significant architectural issues: a `DrizzleClient` type alias defined independently in 12+ files instead of imported from one source, a dual `EventEnvelope` interface definition, a `SearchService` interface mismatch requiring an `as unknown as` cast in production code, a singleton db module that re-exports schemas from domain modules (creating an inverted dependency), and scattered `process.env` access without validation.

- **Description**: At `inngest/index.ts:29-31`, an `emptyRegistry` is created where `getHandler` always returns `undefined`. All pipeline functions (`createCorpusUpdatePipeline`, `createContentAuditPipeline`, `createRegulatoryChangeMonitorPipeline`) are instantiated with this empty registry and exported as `allFunctions`. These are passed directly to `serve()` at `index.ts:170`. Even though the pipeline module initializes a real registry later (at `index.ts:317`), the Inngest functions registered at server startup use the empty one.

## MEDIUM

### [MEDIUM] No environment variable validation at startup
- **Scope**: `joyus-ai-mcp-server/src/` (40+ process.env references)
- **Category**: abstraction | scalability
- **Description**: `process.env` is accessed directly in 40+ locations across `index.ts`, `db/client.ts`, `db/encryption.ts`, `auth/routes.ts`, `tools/executor.ts`, `content/index.ts`, `inngest/client.ts`, `exports/service.ts`, and more. There is no centralized configuration validation at startup. Critical secrets like `TOKEN_ENCRYPTION_KEY` (used in `db/encryption.ts:8`) silently fall through to `undefined` -- the encryption module has a guard but only throws at call time, not at startup.
- **Impact**: The server starts without required configuration and fails only when the misconfigured path is exercised. In production, this means a deployment can appear healthy (health check passes) while token encryption, OAuth, or database operations are broken.
- **Recommendation**: Create a `config.ts` module that validates all required environment variables at import time using Zod (already a dependency). Fail fast at startup with clear error messages listing all missing variables.

### [MEDIUM] Pipeline module uses NodePgDatabase while content module uses ReturnType<typeof drizzle>
- **Scope**: `pipelines/init.ts:34`, `content/types.ts:118`, `index.ts:318`
- **Category**: coupling | abstraction
- **Description**: The pipeline module types its db dependency as `NodePgDatabase` (from `drizzle-orm/node-postgres`), while the content module uses `ReturnType<typeof drizzle>`. These are the same runtime type but TypeScript treats them as incompatible, requiring the cast at `index.ts:318`: `db: db as unknown as import('drizzle-orm/node-postgres').NodePgDatabase`. The same cast appears at `tools/executor.ts:67`.
- **Impact**: Every integration point between modules requires unsafe casts. A Drizzle upgrade that changes internal types could cause silent runtime failures.
- **Recommendation**: Standardize on one canonical type (either `NodePgDatabase` or a custom `DrizzleClient` alias) exported from a shared module, as described in the CRITICAL finding above.

### [MEDIUM] Tool routing relies on string prefix convention with no compile-time safety
- **Scope**: `tools/executor.ts:49-90`, `tools/index.ts:38`
- **Category**: abstraction | scalability
- **Description**: Tool dispatch in `executor.ts` uses string prefix matching (`toolName.startsWith('content_')`, `toolName.startsWith('pipeline_')`, etc.) to route to executors. Tool definitions in `content-tools.ts` and `pipeline-tools.ts` are plain objects with `name: string`, so there is no compile-time guarantee that a tool's name matches the routing prefix. Similarly, `getAllTools` assembles tools by concatenating arrays with no deduplication or conflict detection.
- **Impact**: A tool name typo (e.g., `contents_search` instead of `content_search`) would silently fall through to the "Unknown tool" error. Name collisions between tool sets would be undetected. As the tool catalog grows, the prefix-based routing becomes fragile.
- **Recommendation**: Define tool names as const string literals or enum members with a discriminant (e.g., `namespace: 'content' | 'pipeline' | ...`). Use a tool registry pattern where each executor registers its tools, providing both the definition and handler in one place.

### [MEDIUM] Audit logging in the MCP endpoint is inline and blocking
- **Scope**: `index.ts:210-237`
- **Category**: scalability
- **Description**: Every `tools/call` request writes to the `audit_logs` table synchronously within the request path (lines 210-216 for success, 225-232 for failure). Both the success and error paths `await` the insert before returning the response.
- **Impact**: If the database is slow or under load, every tool call's latency increases by the audit log write time. A database failure during audit logging would cause a tool call to fail even if the tool itself succeeded. At scale, this becomes a bottleneck since every single MCP request requires two database operations (tool execution + audit log).
- **Recommendation**: Fire-and-forget the audit log insert (or use a buffered writer / event bus). The audit log should not be in the critical path of tool response delivery.

## Architecture Health Summary

**Overall: Solid foundation with localized structural debt.**

The two TypeScript packages are cleanly separated -- zero cross-package imports confirmed. Within `joyus-ai-mcp-server`, the domain decomposition into `content/`, `pipelines/`, `auth/`, `scheduler/`, and `tools/` is sound. PostgreSQL schema namespacing (`content`, `pipelines`, `public`) provides good data isolation. Interface-first design is consistently applied for pluggable components (connectors, search providers, entitlement resolvers, voice analyzers, event bus, step handlers).

The most impactful issue is the type system bypass pattern: 12+ copies of `DrizzleClient`, two incompatible `SearchService` interfaces, two `EventEnvelope` definitions, and `NodePgDatabase` vs `ReturnType<typeof drizzle>` -- all culminating in 5+ `as unknown as` casts in production code paths. These are symptoms of a single root cause: the lack of a shared type contract layer between the content, pipelines, and tools modules. Establishing this layer would resolve the CRITICAL and two HIGH findings simultaneously.

The content tool executor bypassing the service layer is the second structural concern. The services exist and are well-designed, but the MCP tool path does not use them. This should be prioritized before adding new content features, as it will compound.

### [CRITICAL] N+1 Query Pattern in ContentRetriever.retrieve()

- **File**: `/Users/AlexUA/claude/joyus-ai/joyus-ai-mcp-server/src/content/generation/retriever.ts:60-78`
- **Category**: query
- **Description**: After running a search that returns N results, the retriever issues one individual `SELECT * FROM content.items WHERE id = ?` query per result in a sequential `for` loop. With `maxSources` defaulting to 5, this is 1 search + 5 sequential DB queries = 6 queries per generation request.
- **Estimated impact**: At 100 concurrent generation requests with 5 sources each: 1 search + 5 item fetches = 600 queries. Each item query is ~2-5ms; sequential execution means 10-25ms per request in DB time alone. At 500 concurrent users this becomes 3,000 queries. The pool (`max: 10`) becomes the bottleneck -- requests queue behind connection acquisition. p95 latency exceeds 500ms budget at roughly 100 concurrent generation requests.
- **Complexity**: O(N) queries per request where N = maxSources (bounded at 5-20 by the caller, so bounded but still unnecessary).
- **Recommendation**: Replace the sequential per-item fetch with a single `SELECT * FROM content.items WHERE id IN (...)` query using Drizzle's `inArray`. This reduces 6 queries to 2 queries per request.

### [CRITICAL] N+1 Query Pattern in DriftMonitor.evaluateRecentGenerations()

- **File**: `/Users/AlexUA/claude/joyus-ai/joyus-ai-mcp-server/src/content/monitoring/drift.ts:55-71`
- **Category**: query
- **Description**: For each profile group, the drift monitor calls `this.analyzer.analyze()` and then issues an individual `UPDATE content.generation_logs SET drift_score = ? WHERE id = ?` for every single generation log -- one UPDATE per log entry, sequentially.
- **Estimated impact**: MAX_BATCH is 100 logs. For a tenant with 5 profiles averaging 20 logs each: 100 individual UPDATE statements per evaluation cycle. Each UPDATE ~2-5ms = 200-500ms total. ### [MAJOR] Sync Engine Upserts Items One-at-a-Time

- **File**: `/Users/AlexUA/claude/joyus-ai/joyus-ai-mcp-server/src/content/sync/engine.ts:222-231`
- **Category**: throughput
- **Description**: Inside `indexInBatches`, items from each connector batch (DEFAULT_BATCH_SIZE = 100) are upserted one at a time in a sequential `for` loop. Each upsert executes a SELECT (to check existence) followed by an INSERT...ON CONFLICT UPDATE -- 2 queries per item.
- **Estimated impact**: For a source with 10,000 items: 10,000 items / 100-per-batch = 100 batches. Per batch: 100 items x 2 queries = 200 queries. Total: 20,000 queries for a full sync. At ~3ms per query = 60 seconds just in DB time. With a freshness window of 1440 minutes (default) and MAX_CONCURRENT_SYNCS = 3, this is sustainable for small sources but will exceed a 10-minute budget at ~3,300 items.
- **Complexity**: O(N * 2) queries where N = total items in source.
- **Recommendation**: (1) Remove the pre-check SELECT -- the `INSERT ... ON CONFLICT DO UPDATE` already handles existence detection; the `existing` check is only used for the return label (`created` vs `updated`), which could be derived from `xmax` in the RETURNING clause. This halves queries to O(N). (2) Batch upserts: group 50-100 items into a single `INSERT ... VALUES (...), (...) ON CONFLICT DO UPDATE`, reducing to O(N/batch) queries.

### [MAJOR] Entitlement Service Issues N+1 Queries for Multi-Product Users

- **File**: `/Users/AlexUA/claude/joyus-ai/joyus-ai-mcp-server/src/content/entitlements/index.ts:184-193` and `213-221`

- **Description**: `getAccessibleSourceIds` and `getAccessibleProfileIds` each issue one query per product ID when the user has multiple products. ## Scalability Analysis

**What breaks first at 10x load (500-5000 concurrent users)?**

1. **Database connection pool (10 connections)** -- The first bottleneck. At 500 concurrent users, with sync jobs holding connections for minutes and each mediation request requiring 6-14 sequential queries, the pool of 10 will be saturated. Connection acquisition will timeout after 2 seconds, causing request failures.

2. **N+1 query patterns in retriever/entitlements** -- At 1000 concurrent mediation messages, the retriever alone generates 5000 individual item queries. Combined with entitlement resolution queries, this is 10,000-14,000 queries/second against a pool of 10 connections.

3. **Scheduler per-task updates** -- At 500 scheduled tasks, the every-minute update cycle consumes 501 queries/minute, holding a pool connection for ~1 second each minute.

**Cost scaling**: Linear with tenants/users for compute and DB queries. The `getDriftSummary` query scales O(profiles * time) per call, which is the only super-linear cost path identified.
