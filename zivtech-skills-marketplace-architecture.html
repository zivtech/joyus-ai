◆ Delivery Channels

Two audiences, two delivery mechanisms, one catalog. Mac developers install skills locally via Claude Code. Windows users access skills remotely through the MCP server.

⚙ Channel 1: Native Claude Code

For Mac developers with Claude Code CLI

  • ‣ Register marketplace once: /plugin marketplace add zivtech-skills
  • ‣ Browse and install: /plugin install audit-website@zivtech-skills
  • ‣ Tool skills (SquirrelScan) run locally on their machine
  • ‣ Knowledge skills inject SKILL.md into Claude's context
  • ‣ Integration skills use shared MCP server or local OAuth
  • ‣ Also supports existing npx skills add installation

☁ Channel 2: Remote MCP

For Windows users via Claude Desktop or Web UI

  • ‣ Connect Claude Desktop to Zivtech AI MCP Server
  • ‣ zivtech_skills MCP tool for catalog browsing
  • ‣ Natural language discovery: "What skills are available?"
  • ‣ Tool skills run server-side (SquirrelScan on the server)
  • ‣ Knowledge skills returned as context via MCP
  • ‣ Web UI at ai.zivtech.com as fallback access method

◆ The zivtech_skills MCP Tool

Windows users interact with the skill catalog through natural language. Claude discovers the zivtech_skills tool via MCP tools/list and uses it when users ask about available skills.

TypeScript — MCP Tool Definition
{
  name: "zivtech_skills",
  description: "Browse and manage Zivtech's curated AI skill catalog. " +
    "Use 'list' to see available skills, 'info <skill>' for details, " +
    "or 'enable <skill>' to activate a skill for your session.",
  inputSchema: {
    type: "object",
    properties: {
      action: {
        type: "string",
        enum: ["list", "info", "enable", "disable", "request"],
        description: "Action to perform"
      },
      skill: { type: "string" },
      reason: { type: "string" }
    },
    required: ["action"]
  }
}

Action Reference

ActionWhat It DoesUser Might Say
listReturns all available skills with descriptions and types"What Zivtech skills are available?"
infoReturns full SKILL.md content, requirements, and example prompts"Tell me about the audit-website skill"
enableActivates a skill's tools and context for the current session"Enable the site audit skill"
disableDeactivates a skill for the current session"Disable the audit skill"
requestCreates a GitHub Issue requesting a new skill"I want to request a skill for PDF form filling"
No special command needed Users don't type "zivtech skill" — they just talk naturally. Claude sees the tool via MCP and routes to it: "audit our website" triggers enable then runs the audit. "What can I do?" triggers list.

◆ Skill Request & Approval Workflow

PMs request skills via GitHub Issues or through Claude. Devs review, build, and submit via PR. CI validates. Merge deploys the skill to both channels.

1

PM Requests a Skill

Opens a GitHub Issue using the "Skill Request" template, or tells Claude "I want to request a skill for X" which auto-creates the issue via the request action.

2

Triage & Assignment

Dev lead evaluates the request for security implications, usefulness, and maintenance burden. If approved, assigns to a developer.

3

Development

Dev creates a skill/<name> branch, writes SKILL.md + plugin structure, tests locally with Claude Code, tests remotely for server-side tools.

4

Review (PR)

PR opened using the "New Skill" template. CI validates SKILL.md format, marketplace.json syntax, scans for secrets, and checks plugin structure. Reviewer checks instruction quality and security.

5

Deployment

PR merged = skill approved. marketplace.json updated. GitHub Action notifies MCP server to refresh catalog. Mac devs run /plugin marketplace update. Windows users see new skills automatically.

CI Validation (validate-skill.yml)

GitHub Actions — runs on PRs touching plugins/**
# What the CI checks:
1. marketplace.json syntax & required fields
2. Each plugin has unique name, valid source
3. SKILL.md files have YAML frontmatter (---)
4. SKILL.md files have description field
5. No secrets detected (API keys, tokens)
6. Plugin directory structure is valid

marketplace.json Structure

The marketplace.json wraps both existing skills (via GitHub source references) and new plugins (via relative paths). Existing skills use strict: false so they don't require plugin.json.

.claude-plugin/marketplace.json (abbreviated)
{
  "name": "zivtech-skills",
  "owner": { "name": "Zivtech", "email": "dev@zivtech.com" },
  "plugins": [
    // New plugins — relative path source
    { "name": "audit-website", "source": "./plugins/audit-website", ... },
    { "name": "zivtech-jira", "source": "./plugins/zivtech-jira", ... },
    { "name": "zivtech-confluence", "source": "./plugins/zivtech-confluence", ... },

    // Existing skills — GitHub source + skills path override
    { "name": "zivtech-ticket-writing",
      "source": { "source": "github", "repo": "zivtech/zivtech-claude-skills" },
      "skills": "./.claude/skills/zivtech-ticket-writing",
      "strict": false },
    // ... same pattern for all existing skills
  ]
}

When Windows users invoke tool skills like SquirrelScan, the CLI runs on the Zivtech server. The MCP server validates, executes, and returns results.

◆ Open Questions

Decisions that need to be made as implementation progresses.

Repo Visibility: Public or Private?

A private repo means all users need GitHub access. Public means skills are visible to anyone but easier to distribute. Could use a private repo with deploy tokens for the MCP server.

Skill Catalog Auto-Refresh

When a skill is merged, how does the MCP server know? Options: GitHub webhook on merge triggers server refresh, or periodic polling (every 5 min). Webhook is more responsive but requires endpoint exposure.

Per-User Skill Enablement

Do all users get all skills, or should there be user-level or role-level toggles? Could start with "everyone gets everything" and add granularity later.

Server-Side Cost Implications

Tool skills like SquirrelScan add server CPU and network usage. Need rate limiting (e.g., max 5 audits per user per hour). Consider queuing for expensive operations.

Skill Versioning & In-Flight Sessions

When a skill is updated in the repo, do active sessions keep the old version or get the new one? Suggest: sessions keep their version until next enable, new sessions get latest.

Claude Desktop Extension for Windows

Claude Desktop supports custom MCP servers via config. Need to document the exact setup for Windows users connecting to the Zivtech server, including auth token provisioning.
